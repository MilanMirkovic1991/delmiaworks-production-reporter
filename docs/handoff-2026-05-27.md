## TL;DR (one paragraph)

We're building a wizard app on top of the DelmiaWorks (IQMS) WebAPI. The Phase-1 read-only flow (item → SO → releases → BOM) is shipped. We've since extended into **PO creation + auto-receive** on top of WO tree expansion. As of this session, the receive flow runs end-to-end against the real DW VM. Out of a 111-line PO test, **56 receipts succeeded** and **55 failed** — but every failure is a DW-side master-data issue (no recipe card, orphan master labels, serialized + fractional qty, transient currency lookup), **not** a bug in our code. The two real bugs we fixed this session were:

1. **PO_RELEASES.Seq was never set** → added follow-up `UpdatePOReleaseItem` per release, with a per-arInvtId Seq counter.
2. **MASTER_LABEL.SERIALNO was sent as `'1'` for every receipt** → ORA-00001 on `AK_MASTER_LABEL_SERIAL` killed every receipt after the first. Now we read max(SERIALNO) from `/Labels/PrintLabel/MasterLabels/0` once per receivePO run and allocate 7-digit zero-padded sequential serials forward.

## What this session delivered (in order)

### 1. `feat(po): set PO_RELEASES.Seq per arInvtId via UpdatePOReleaseItem` — commit `59ff039`

**Problem.** User noticed `PO_RELEASES.Seq` was `0` for every release we created. Spec: per-arInvtId counter (1, 2, 3, …) within a PO ("redni broj za svaki PO_RELEASES po artiklu sa PO").

**Discovery.** Probed `/Help/Api?apiId=POST-POReceiving-PO-CreatePOReleaseItem-id_poDetailId_quantity_requestDate_promiseDate`. The signature has **no `Seq` parameter**. But `UpdatePOReleaseItem` accepts a full body that includes `Seq`. So the pattern is: Create → Update.

**Implementation** (`backend/src/dwClient/po.ts`, inside `createPurchaseOrder`):

```typescript
const seqByArInvt = new Map<number, number>();
// after each successful CreatePOReleaseItem:
const nextSeq = (seqByArInvt.get(it.arInvtId) ?? 0) + 1;
seqByArInvt.set(it.arInvtId, nextSeq);
try {
  await http.post(`/POReceiving/PO/UpdatePOReleaseItem/${releaseId}`,
    { ...(releaseBody as Record<string, unknown>), Seq: nextSeq });
} catch (seqErr) { /* non-fatal: release exists, log warn */ }
```

**Result type** got a new optional `seq?: number` on `POLineItemResult`. Test added: `createPurchaseOrder: PO_RELEASES.Seq counts per arInvtId (1, 2, 3...) within the same PO`.

### 2. `fix(receive): global Serial counter, 7-digit padded, never duplicates` — commit `3c17016`

**Problem.** Live test failed on every receipt past the first with:

```
ORA-00001: unique constraint (IQMS.AK_MASTER_LABEL_SERIAL) violated
   at IQMS.DAL.Lib.POReceiving.PORepository.CreateMasterLabelRow
```

Root cause: we hard-coded `Serial: '1'` in the `CreatePoReceiptsLabelsPlan` body. Every receipt tried to insert `MASTER_LABEL.SERIALNO = '1'`. First win, every subsequent attempt collides.

User spec, verbatim: *"polje SERIALNO u tabeli MASTER_LABEL treba da ima 7 cifara - 0000001, pa onda 0000002... I nikada ne sme da se duplira. […] koristiš brojač 0000001, 0000002 itd, bez obzira na artikal. Broji do u nedogled."*

So: **one global counter, 7-digit zero-padded, monotonic forever, scoped across the whole MASTER_LABEL table.**

**Discovery (took two probe rounds).**
- First-try `/Help/Api?apiId=GET-POReceiving-PO-MasterLabel-id` returned 500 (endpoint doesn't exist under that name).
- Hit `/Help` and parsed the HTML for all `apiId=...Controller` references. Got the controller catalog.
- Drilled into `WebAPI.Areas.Labels.Controllers.PrintLabelController` → found `GET-Labels-PrintLabel-MasterLabels-id`. Calling `/Labels/PrintLabel/MasterLabels/0` returns **every MASTER_LABEL row** with a `Serial` field. Perfect for a max-scan.

**Implementation** (`backend/src/dwClient/po.ts`, top of `receivePO`):

```typescript
let serialCounter = 0;
try {
  const r = await http.get(`/Labels/PrintLabel/MasterLabels/0`);
  const rows = Array.isArray(r.data?.data ?? r.data) ? (r.data?.data ?? r.data) : [];
  for (const ml of rows) {
    const n = Number.parseInt(String(ml.Serial ?? '').trim(), 10);
    if (Number.isFinite(n) && n > serialCounter) serialCounter = n;
  }
} catch { /* non-fatal; start from 0 */ }
```

And inside the receipt loop:

```typescript
serialCounter++;
const serialNo = String(serialCounter).padStart(7, '0');
const prepBody = { POReceiptsId: poReceiptId, LabelsCount: 1, Qty: qty, Serial: serialNo };
```

**Surfaced through the stack.** `ReceiptResult.serialNo?: string` propagates to `backend/test/dwClient/po.test.ts` mocks, `frontend/src/api/client.ts` type, and the **"Detalji prijema" table** on `frontend/src/pages/WorkOrders.tsx` (new "Serijski br." column rendering `<code>{r.serialNo}</code>`).

**Tests.** All 7 existing `receivePO` tests got the new `nock(BASE).get('/Labels/PrintLabel/MasterLabels/0').reply(200, { data: [] })` mock. New test `receivePO: Serial counter continues from max existing MASTER_LABEL.Serial + 1` exercises:
- max parsing across rows with mixed numeric/non-numeric/null `Serial`
- counter starting at max+1
- 7-digit zero-padding
- multi-receipt monotonic increment

Missing mandatory field 'Lot #'. [Item# 095725210, Loc: 1A]. Master Label ID = 254
Missing mandatory field 'Lot #'. [Item# 303688109, Loc: 1A]. Master Label ID = 327

Note `Master Label ID = 254` and `327` — **these are orphan MASTER_LABEL rows from earlier failed runs** (this run's MasterLabelIds are ≥ 568). ### C) `3 receipts` — "Serialized inventory + fractional qty"

```
This is a serialized inventory controlled item - total qty must match total labels qty
```

| `arInvtId` | `qty` |
|---|---|
| 191805 | 12.8 |
| 191716 | 2.1472 |
| 191682 | 32.0176 |

Serialized Inventory Control wants integer-divisible label counts. 2.1472 can't be split into integer labels. Three options for the user:
1. Round those quantities up/down to integers at PO-create time.
2. Switch those specific items out of Serialized Inventory Control in DW master data.
3. Accept manual receipt in DW UI for these items.

### D) `3 receipts` — "PO Receipt could not be found while determining currency"

```
ERROR EXECUTING TRANSACTION [invtrans.trans_batch:NN] -
  The Purchase Order Receipt could not be found while determining currency.
```

`poReceiptId` 653, 671, 676. DW just returned these IDs from `CreatePOReceipt`, then can't find them when the Post step tries to look up currency. Looks like a transient race or vendor-currency config gap. A simple retry-button on the failed rows would likely clear these.

## DW WebAPI knowledge gained (don't re-derive next session)

### Probing pattern

```typescript
// Get DTO sample for any endpoint
const r = await http.get('/Help/Api?apiId=METHOD-Area-Controller-Action-id_param1_param2');
// → returns HTML with <pre> blocks containing JSON / XML sample bodies

// Get controller catalog
const r = await http.get('/Help');
// → HTML; grep for `apiId=([A-Za-z0-9._]+Controller)`
// → then `/Help/Controller?apiId=<full.controller.path>` for actions list
```

The probe scripts we used were temporary and deleted on commit. If next session needs them again, the recipe is: write a tiny `tools/peek-*.ts`, run with `npx tsx tools/peek-foo.ts`, delete after the discovery is in code+comments.

### Confirmed DTO schemas

**`POReceiptsLabelsPlan`** (body for `POST /POReceiving/PO/CreatePoReceiptsLabelsPlan/0`):
```json
{ "POReceiptsId": number, "LabelsCount": number, "Qty": number, "Serial": "string" }
```
Pitfalls: `POReceiptsId` (capital PO, trailing s), `LabelsCount` (not LabelCount), `Serial` is a string. `LotNo` / `ArInvtId` / `LmLabelsId` do **not** belong here.

**`ReceivingTransSettings`** (body for `POST /POReceiving/PO/PostPOReceiptAndUpdateMasterLabel/0?poReceiptId={id}`):
```json
{ "UseDefaultLocation": true, "LocationId": 0, "LotNo": "string", "TransDate": "iso" }
```
`LotNo` lives here, not in the Plan body.

**`PO_RELEASES` row** (`GET /POReceiving/PO/POReleaseItem/{id}` and `POReleaseItems/0?poLineItemId=...`):
```json
{ "Id": n, "PODetailId": n, "Quantity": f, "RequestDate": "iso",
  "PromiseDate": "iso", "Seq": n, "DropShipReleasesId": n }
```
`CreatePOReleaseItem` does NOT accept `Seq` as a query param. Use `UpdatePOReleaseItem/{id}` with full body to set it.

**`MasterLabel` row** (`GET /Labels/PrintLabel/MasterLabels/0`):
Returns array of master labels with many fields. Key ones we use: `Id`, `Serial`, `Arinvt[Id]`, `LotDate`, `LmLabelsId`. **`Serial` is the value we generate** (7-digit padded string).

### Discovered controllers (`/Help/Controller?apiId=...`)

Useful ones we've poked at:
- `WebAPI.Areas.Labels.Controllers.PrintLabelController` — `MasterLabels`, `MasterLabel`, `Labels`, print actions
- `WebAPI.Areas.Inventory.Controllers.SerialTrackingController` — `SerialItems`, `ItemBySerial`
- `WebAPI.Areas.Inventory.Controllers.DispositionController` — `MasterLabel`, dispo flows
- `WebAPI.Areas.Inventory.Controllers.ScanIdController` — `SerialMasterLabel`, reprint

- `WebAPI.Areas.POReceiving.Controllers.POController` — every PO/receipt action we already use
